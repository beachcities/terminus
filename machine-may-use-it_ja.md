# 機械可読・機械理解可能性の先にあるもの：機械が「扱ってよいか」

**版**: 2026-06-13
**位置づけ**: AI時代のデータガバナンスに関する技術構想。データ・コード・モデルが連動する環境で、文書に「扱ってよいかのルール」をどう持たせ、どう強制するかを整理する。具体的な実装・適用は別文書で扱う。

---

## 1. 背景：機械可読の先にある「機械が扱ってよいか」

データ記述フォーマットは、「構造の表現力」と「人間・機械双方の可読性・安全性」を順に獲得してきた。

- **CSV** — 平面的な記録。階層も文脈も表現できない。
- **JSON** — 機械向けの階層構造。表現力は高いが、目視での品質管理に難がある。
- **YAML（特に Strict YAML）** — 構造と人間可読性を両立し、解釈のブレを抑えて決定論的な扱いやすさを得た。

ここまでは「データをいかに正確・安全に**記述**するか」、すなわち機械可読性と、その先の機械理解可能性をめぐる進化だった。しかしデータ・コード・モデルが密接に連動する現在、次に必要なのは別の軸である——「そのデータを、機械（AI）が**どう扱ってよいか**」というルールのレイヤー（ガバナンス）である。

機械可読（読める）から機械理解可能（意味がわかる）へ、そしてその先の「扱ってよいか（許諾・ポリシー）」へ。本稿はこの最後の軸を扱う。

**DocLang**（LF AI & Data Foundation、2026年6月に仕様策定WG発足）は、非構造化文書に対し、構造・セマンティクス・レイアウトの保持に加えて、このガバナンスを仕様レベルで統合したAIネイティブな文書フォーマットである。

なお、上記の世代論は「フォーマットが順に獲得してきた性質」の整理であり、技術系譜ではない点に注意する。**DocLangはYAMLの後継ではなく、XMLベース**である（`.dclg.xml`、検証はXSD＋Schematron、`pip install doclang`）。また構造化データ記述の置き換えでもなく、対象は非構造化文書である。

### 1.1 適材適所のアーキテクチャ（基本方針）

本構想は、データの性質に応じた役割分担を前提とする。

- **構造化データ**（API連携、エージェントの状態管理など）→ **Agentic Data（YAML等）**
- **非構造化文書**（人間とAIが共有するドキュメント）→ **DocLang**

両者は競合せず、対象が異なる。この前提を最初に固めておくことで、「YAMLで足りるのでは」という問いに正しく答えられる。

## 2. 課題：なぜLLMへのプロンプトだけでは不十分か

AIエージェントにデータ処理を委ねる際、自然言語のプロンプト指示のみに依存することには、アーキテクチャ上の限界がある。

1. **構造の破壊** — LLMは文書をフラットな文字列として扱いがちで、元の階層やスキーマを意図せず壊す。
2. **ガバナンスの忘却** — 「個人情報は除外」とシステムプロンプトに書いても、ハルシネーションやプロンプトインジェクションで突破され得る。
3. **来歴の喪失** — 丸ごとの書き換えにより、なぜ・どこを直したかのトレーサビリティが失われる。

### 2.1 既存の手段が届く範囲と、届かない範囲

データ品質の世界には、すでに確立された手段がある。それぞれが担う範囲を整理すると、抜けている層が見えてくる。

- **ルールベースの検証**（構造・型・整合性を決定論的にチェックする方式）が担うのは「**データそれ自体の正しさ**」である。妥当で必要だが、利用の可否は射程外。
- **データカタログ記述の標準**（DCAT 等）が担うのは「**何のデータかの記述**」である。汎用的だが、これも「そのデータに何をしてよいか」を操作の前に強制する層ではない。

つまり「データの正しさ」と「カタログ記述」は手段が揃っているが、「**そのデータを、誰が・どの操作で・どこまで扱ってよいか**」を機械的に強制する層が抜けている。そして、この層をプロンプトに書くだけで代替することはできない。プロンプトは確率的推論への入力であり、上記3つの限界（破壊・忘却・喪失）に晒されるからである。これが「なぜプロンプトだけでは不十分か」への答えである。

## 3. DocLangのガバナンスメタデータ（仕様 v0.4.0）

DocLangは「そのデータをどう扱ってよいか」を、文書の `<head>` に機械可読な**宣言**として持つ。仕様で定義されている主な操作別コントロールは次の通り。

| メタデータ | 制御内容 |
|---|---|
| `extraction_permitted` | 抽出の可否 |
| `pii_extraction_allowed` | 文書全体は抽出可でも、氏名・住所等の個人情報部分だけ抽出を止められる |
| `rag_indexing_allowed` | RAGのインデックスに取り込んでよいか（＝その文書をRAGの回答素材として取り込んでよいか） |
| `training_permitted` | モデル学習への利用可否 |

このほか Licensing / Data classification / Retention / Compliance frameworks や、PII関連の詳細属性（`pii_status`、`ai_use_restriction` 等、GDPR・ISO 27701等との対応づけ）が定義されている。

重要な性質が2つある。

- **部位ごとの上書き** — 文書レベルで宣言したうえで、コンポーネント（部位）ごとにオーバーライドできる。「この表は抽出可、この段落のPIIは不可」といった細粒度の制御ができる。
- **DCATとの補完関係** — DCATが「何のデータか」を記述するのに対し、DocLangは「そのデータに何をしてよいか」を記述する。置き換えではなく、別の軸を足す。

### 3.1 宣言と強制の分離（設計上の要）

DocLangが定義するのはマシンリーダブルな「**宣言**」であって、実行を制御する「**強制機構（エンフォースメント）**」ではない。宣言をプロンプトに貼るだけなら、結局「お願い」にとどまる。

したがって、宣言を確実に読み取り、LLMの確率的推論の**前に**決定論的な検査・遮断を行う層を、こちら側で実装する必要がある。これが次章のスキルの核心であり、本構想全体の要である。

---

## 4. データスチュワードが持つべきスキル

データが、どう扱われることを許すか。その振る舞いを、操作の前に決定論的に規定できるようにする。そうすることで、これまで利活用しきれなかった非構造化データの価値まで、安心して引き出せるようになる。それを支える、データスチュワードが持つべきスキルが、次の三つである。

1. **構造・セマンティクス不整合検知** — 構造レイヤーを決定論的に検証し、期待データモデル（オープンデータ仕様等）とのズレを、構造を壊さずに検知・修正提示する。
2. **ガバナンス・使用ポリシー監査** — `<head>` のガバナンス宣言を機械的に解釈し、操作の前に可否を判定する。プロンプトによる「お願い」ではなく、データに紐づくルールとして強制力を持たせる。
3. **コンテキストを維持した正規化** — マルチレイヤーを保ったまま、表記揺れ等の特定部分にのみ非破壊パッチを当て、文脈と修正来歴を保持する。

これらはいずれも「文書に紐づいた宣言を読んで、操作の前に決定論的に振る舞いを決める」という共通の形を持つ。ルールを公表したうえで横断的に再利用できる部品として共有・流通させれば、車輪の再発明を避け、集合知として循環させ得る知的基盤になる。スキルの分類体系そのものは今後の課題である。

## 5. 現物：ガバナンス・コア（Sphragis）

上記スキルの核——ガバナンス宣言を読んで操作の前に可否を返す部分——は、タスク非依存・フレームワーク非依存の独立OSSとして最小実装を公開している。

- **リポジトリ**: https://github.com/beachcities/sphragis （Apache-2.0、コアは依存ゼロ）
- **できること**: DocLangの `<head>` を解釈し、操作（extract / rag_index / rag_retrieve / train / share_downstream）に対して `allow` / `allow_with_obligations` / `deny` を決定論的に返す。義務（変換要求・監査・human-in-the-loop）も併せて提示する。
- **既定の姿勢**: strict by default（明示的に許可されていない操作は拒否）。これは「禁止されていなかったからやってしまった」という型の漏えいを、コードレベルで塞ぐための設計である。
- **デモ**: `demo/` にインタラクティブなシミュレーターを同梱。「宣言を読んで操作の前に機械的に遮断する」が何かを触って確認できる（PIIを含む抽出をStrictで試すのが分かりやすい）。

「ルールを公表し、横断的に使える形で流通させる」という本構想の方針を、この小さなOSS自体が体現している。中心にあるのは一つのテーゼである——**宣言は強制機構ではない。プロンプトは要求であって、ルールではない（A prompt is a request, not a rule）**。
